# Performance and Feature Improvement Requests

Customers sometimes report that the application is slow or ask for improvements to existing features. This document distinguishes performance bugs from feature requests and covers how to handle each.

## Performance issues vs feature requests

- **Performance issue:** The feature exists and works but is unacceptably slow. → Route as `product_bug` with priority based on severity.
- **Feature request:** The customer wants the product to do something it does not currently do at all. → Route as `feature_request`.
- **UX improvement:** The feature works but the customer wants it reorganised, renamed, or made more accessible. → Route as `feature_request`.

## Collecting performance data

For genuine performance issues:
- Ask for the specific action that is slow (e.g. "bulk export of 10,000 rows takes 20 minutes")
- Ask for the approximate data volume involved
- Ask if it has always been slow or started recently (regression vs. scale issue)

## How requests are prioritised

Product prioritises feature requests by:
1. Number of customers requesting the same thing
2. Alignment with strategic roadmap
3. Implementation cost vs. customer impact

Customers should be aware their request may not be implemented even if popular.
