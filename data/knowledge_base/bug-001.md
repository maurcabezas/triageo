# Crash Reporting and Bug Triage Process

When a customer reports a crash or severe bug, follow this process to gather enough information for the engineering team to reproduce and fix the issue.

## Information to collect

- App version number
- Operating system and version
- Device model (for mobile) or browser and version (for web)
- Steps to reproduce the crash
- Frequency (always, sometimes, once)
- Error message or crash log if available
- Screenshots or screen recording

## Severity classification

| Severity | Description | SLA |
|---|---|---|
| P0 — Critical | App unusable, data loss possible | Engineering notified immediately |
| P1 — High | Core feature broken, no workaround | Fix within 48 hours |
| P2 — Medium | Feature broken but workaround exists | Fix within 2 weeks |
| P3 — Low | Minor visual or cosmetic issue | Next release cycle |

## Creating an engineering ticket

Once information is collected, create a bug report in the issue tracker with:
- Reproduction steps
- Environment details
- Customer impact (number of users affected)
- Attach any logs or screenshots

KEYWORDS: crash, bug, error, exception, null pointer, app not working, broken, freeze, hangs, not responding, stack trace, crash report, glitch
